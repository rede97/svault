`svault-core` exposes a `cli` feature (optional dep on `clap`) so that `HashAlgorithm` etc. can derive `ValueEnum` when needed by the CLI.

## Architecture

Svault is a content-addressed multimedia archival tool. A vault is any directory that contains `.svault/vault.db` (SQLite) and `svault.toml`. The CLI walks up from CWD to discover the vault root.

### Import pipeline (`import/mod.rs`)

The only fully-implemented command is `import`. It runs five stages:

| Stage | What happens |
|-------|--------------|
| A | VFS walk of source dir — collects `ScanEntry` list filtered by `allowed_extensions` |
| B | CRC32C fingerprint of each file probed against DB cache → classifies as `LikelyNew` or `LikelyCacheDuplicate` |
| C | Copy `LikelyNew` files into vault under a temp staging path using best `TransferStrategy` |
| D | Strong hash (xxh3_128 or sha256, per config/flag) + three-layer dedup check |
| E | Batch DB write (event + materialised-view row) + manifest file written to `.svault/manifests/` |

Stage B uses `crc32c_epoch` to invalidate the probe cache when the hash algorithm changes.

### Database design

The DB is **event-sourced**: all state changes are appended to the `events` table (with a SHA-256 tamper-evident hash chain). The `files`, `assets`, `media_groups`, and `derivatives` tables are materialised views rebuilt by replaying events. `db verify-chain` / `db replay` (stubbed) will validate and rebuild these views.

File identity uses a two-tier hash:
- **xxh3_128** — fast non-cryptographic; used as temporary identity and for `verify --fast`
- **sha256** — cryptographic; computed lazily on collision or background-hash; takes precedence when present

### VFS layer

`VfsBackend` is a trait abstracting filesystem operations (list, walk, copy, open_read, create_dir_all). `SystemFs` is the only current implementation. `TransferStrategy` is negotiated between source and destination capabilities: Reflink > Hardlink > StreamCopy.

### Configuration (`svault.toml`)

Lives at the vault root. Key fields:

```toml
[global]
hash = "xxh3_128"        # or "sha256"
sync_strategy = "auto"   # or "reflink" | "hardlink" | "copy"

[import]
path_template = "$year/$mon-$day/$device/$filename"
rename_template = "$filename.$n.$ext"
allowed_extensions = ["jpg", "heic", "cr3", "mov", …]
```

## Testing

### Integration test framework (`tests/`)

The test suite is Python-based and uses a **tmpfs RAMDisk** for full isolation — all vault operations happen in `/tmp/svault-ramdisk`, nothing is written to the source tree.

```
tests/
├── .venv/                     # Python venv (Pillow + piexif); never commit
├── fixtures/
│   ├── source/                # Synthetic JPEGs generated by gen_fixtures.py
│   ├── chaos/                 # Edge-case files (truncated, moved, renamed)
│   └── test_rules.json        # Expected outcomes per scenario
├── gen_fixtures.py            # Generates test JPEGs with EXIF metadata
├── run_tests.py               # Test runner (RAMDisk mount, import, validate)
└── check_results.py           # Standalone result validator (reads DB + logs)
```

**Run the full test suite:**
```bash
# First-time setup
python3 -m venv tests/.venv
tests/.venv/bin/pip install Pillow piexif

# Run tests (requires sudo for tmpfs mount)
tests/.venv/bin/python3 tests/run_tests.py

# Options
tests/.venv/bin/python3 tests/run_tests.py --no-build   # skip cargo build
tests/.venv/bin/python3 tests/run_tests.py --keep       # keep RAMDisk for inspection
tests/.venv/bin/python3 tests/run_tests.py --chaos      # include chaos scenarios
tests/.venv/bin/python3 tests/run_tests.py --verbose    # per-check detail
```

**Critical rule: never run `svault init` or `svault import` from within the project directory.** All tests must operate on the RAMDisk vault at `/tmp/svault-ramdisk/vault/`. The `svault` binary uses CWD-based vault discovery — running it from the project root will find (or create) `.svault/` inside the source tree.

**Vault discovery gotcha:** `svault init` ignores `--vault` and always initialises in CWD. Always `cd` into the target directory or run with `cwd=` when invoking via Python subprocess.

## Key design decisions

- **No delete command** — Svault never removes files. After import, users review the manifest and delete source files manually.
- **Vault discovery** — `find_vault_root()` walks up from CWD (or `--vault` override), enabling `svault import` from any subdirectory.
- **Parallel hashing** — Stage D uses `rayon` for parallel strong-hash computation across `LikelyNew` files.
- **`clap` is optional in `svault-core`** — gated behind the `cli` feature flag to keep the library dependency-light.
