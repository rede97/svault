# Import Pipeline Design

> 导入流水线架构设计方案

---

## 设计目标

- 以最小 IO 代价完成文件导入，避免重复传输
- 在文件复制发生之前发现重复，而非之后
- SHA-256 惰性计算：无碰撞时不计算，不浪费 IO
- 数据库写入不成为并发瓶颈
- 任何情况下不写入脏数据（多层安全网）

---

## 流水线架构（Pipeline Model）

导入过程按阶段拆分为流水线，各阶段通过 channel 连接，独立并发执行：

```
┌─────────────────────────────────────────────────────────┐
│  Stage A: 目录扫描（1 线程）                             │
│  递归遍历源目录，产出文件路径列表                        │
└──────────────────────┬──────────────────────────────────┘
                       │ channel
┌──────────────────────▼──────────────────────────────────┐
│  Stage B: 指纹计算线程池（N 线程，IO 密集）              │
│  读取格式声明的字节区间（头部/尾部 64KB）                │
│  计算 CRC32C + 解析 EXIF 关键字段                        │
│  查数据库缓存：命中则标记 skip，不进入 Stage C           │
└──────────────────────┬──────────────────────────────────┘
                       │ channel（仅未命中缓存的文件）
┌──────────────────────▼──────────────────────────────────┐
│  Stage C: SHA-256 计算线程池（M 线程，IO 密集）          │
│  读取完整文件，计算 SHA-256                              │
│  查内存 DashMap（本批次去重）                            │
│  查数据库（跨会话去重）                                  │
│  通过：insert DashMap，进入文件复制                      │
│  重复：标记 duplicate，跳过复制                          │
└──────────────────────┬──────────────────────────────────┘
                       │ channel（仅通过去重的文件）
┌──────────────────────▼──────────────────────────────────┐
│  Stage C2: 文件复制（与 Stage C 同线程池）               │
│  执行实际文件传输（reflink / hardlink / stream）         │
└──────────────────────┬──────────────────────────────────┘
                       │ channel
┌──────────────────────▼──────────────────────────────────┐
│  Stage D: 数据库写入（1 线程，批量提交）                 │
│  消费上游所有结果（imported / skipped / duplicate）      │
│  批量写入，sha256 唯一约束兜底                           │
└─────────────────────────────────────────────────────────┘
```

### 为什么 Stage B 和 Stage C 分开？

- Stage B 每次只读 64KB，大量并发无压力
- Stage C 读完整文件，并发过高导致磁盘随机 IO 剧烈，HDD 场景下反而比串行慢
- 分开可以为两个阶段独立调节并发数

### SHA-256 惰性计算

SHA-256 是**惰性的**——只有在 Stage B 指纹发生碰撞时才会触发计算。如果一批导入的文件全部是新文件（指纹无碰撞），SHA-256 计算量为零。

```
正常导入（无碰撞）：
  Stage B 指纹 → 无碰撞 → 直接复制 → 写入数据库（sha256 = NULL）

指纹碰撞时（CRC32C + EXIF 相同）：
  立即触发 Stage C → 计算 SHA-256 → 精确确认是否真的重复

后台补全任务（系统空闲时）：
  扫描 sha256 = NULL 的记录 → 计算 SHA-256 → 更新数据库
```

`sha256 = NULL` 的语义：文件已安全导入，内容身份待后台确认。不影响正常浏览和使用，仅影响跨会话的精确去重查询。后台补全任务可通过 `svault background-hash` 手动触发，或在系统空闲时自动运行。

## 并发去重：三层安全网

### 问题场景

两个线程独立读取了内容相同的两个文件，各自完成 SHA-256 计算后同时准备写入：

```
线程 A：IMG_001.CR3     → SHA-256 = abc123 → 准备写入
线程 B：IMG_001_copy.CR3 → SHA-256 = abc123 → 准备写入
                                    ↑ 竞态：两者都认为自己是新文件
```

### 三层安全网

#### 层 1：内存 DashMap（同批次去重，阻止文件复制）

```
Arc<DashMap<SHA256, FileInfo>>
```

- Stage C 完成 SHA-256 后，执行原子性 `insert_if_absent`
- 先到的线程 insert 成功 → 进入文件复制
- 后到的线程发现已存在 → 立即标记 duplicate，**跳过文件复制**
- 使用 `DashMap`（分片锁）而非 `Mutex<HashMap>`，避免单点热点
- 内存占用：10 万文件 × 32 字节 = ~3MB，可忽略

**作用**：发现同一批次内的重复，在 IO 发生前拦截。

#### 层 2：数据库查询（跨会话去重，阻止文件复制）

```sql
SELECT id FROM files WHERE sha256 = ? LIMIT 1
```

- Stage C 完成 SHA-256 后，在 insert DashMap 之前先查一次数据库
- 命中：历史导入中已存在该文件 → 标记 duplicate，**跳过文件复制**
- 未命中：继续 insert DashMap 流程
- 此查询为只读，可在 Stage C 线程中直接执行（SQLite WAL 模式支持并发读）

**作用**：发现跨会话的重复（上次导入过的文件），在 IO 发生前拦截。

#### 层 3：数据库唯一约束（最终兜底，防止脏数据）

```sql
CREATE UNIQUE INDEX idx_files_sha256 ON files(sha256);
```

- 即使层 1 和层 2 因极端竞态条件漏掉了重复，数据库拒绝插入
- 写入线程捕获 `UNIQUE constraint failed`，将该条记录补标为 duplicate
- **任何情况下不会产生重复的数据库记录**

**作用**：兜底保障，不依赖上层逻辑的正确性。

### 完整去重决策流程

```
Stage C: 计算 SHA-256 完成
    │
    ├─ 查数据库：历史记录存在? ──→ duplicate，跳过复制
    │
    ├─ insert DashMap（原子）：本批次已存在? ──→ duplicate，跳过复制
    │
    └─ 两者均无 → 进入文件复制
                        │
                        └─ 复制完成 → 写入 Stage D channel

Stage D: 批量写入数据库
    └─ UNIQUE 冲突（极端竞态）→ 补标 duplicate，不报错
```

## 数据库写入策略

### 为什么批量写入

SQLite 每次事务提交触发 `fsync`（刷盘），逐条写入的性能约为 100-500 条/秒。
批量写入（单事务多条）可达 10,000-50,000 条/秒，差距 100 倍以上。

### 批量提交触发条件（满足任一即提交）

```
- 缓冲区积累达 500 条记录
- 距上次提交超过 2 秒
- 上游所有 channel 关闭（导入结束，强制 flush）
```

### 写入时序

| 时机 | 写入内容 | 说明 |
|------|----------|------|
| Stage B 完成 | `crc32c_hdr`, `exif_fp` | 更新指纹缓存，供下次扫描复用 |
| Stage C 完成 | `sha256` | 更新 SHA-256 缓存 |
| 文件复制完成 | `status=imported`, `dest_path` | 记录最终导入结果 |
| duplicate 判定 | `status=duplicate`, `duplicate_of` | 记录重复关系 |

所有写入通过单一 Stage D 线程序列化执行，避免 SQLite 写锁竞争。
读操作（层 2 查询）在各 Stage C 线程中直接执行（WAL 模式下并发读安全）。

## 并发数参考

| 阶段 | 本地 SSD | 本地 HDD | 网络 SMB/MTP |
|------|----------|----------|-------------|
| Stage B（64KB 指纹） | 8–16 | 4–8 | 16–32 |
| Stage C（SHA-256 + 复制） | 4–8 | 2–4 | 4–8 |
| Stage D（DB 写入） | 1 | 1 | 1 |

- 网络场景 Stage B 并发数更高：IO 延迟大，多线程掩盖等待时间
- HDD 场景 Stage C 并发数更低：顺序读优于并发随机读
- 并发数可配置，默认值根据 VFS `capabilities()` 声明的存储类型自动选择

## 导入结果清单（Manifest）

导入完成后输出映射清单，供用户核查后手动删除源文件：

```
# svault-import-manifest-20240315T143000.txt
# Review this file. Delete source files manually after verifying the archive.

IMPORTED   /archive/2024/03/15/IMG_001.CR3   <--  /mnt/card/DCIM/IMG_001.CR3
IMPORTED   /archive/2024/03/15/IMG_001.JPG   <--  /mnt/card/DCIM/IMG_001.JPG
DUPLICATE  (sha256:abc123, existing: /archive/2023/12/01/IMG_001.CR3)  <--  /mnt/card/DCIM/IMG_002.CR3
SKIPPED    (cache hit, no change)             <--  /mnt/card/DCIM/IMG_003.JPG
```

Svault 不删除任何源文件。清单是用户与工具之间的契约：工具报告发生了什么，用户决定下一步。
